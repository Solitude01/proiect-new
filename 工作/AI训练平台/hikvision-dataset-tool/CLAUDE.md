This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

```bash
# Install dependencies
pip install -r requirements.txt

# GUI mode (recommended)
python main.py --gui

# CLI auto mode (reads dataset info from current browser page)
python main.py --auto --output ./my_dataset

# CLI manual mode
python main.py --dataset <id> --version <id> --token "<token>"

# Export downloaded dataset to COCO format
python main.py --export-coco ./dataset_xxx_timestamp
```

There is no test suite, linting, or type-checking configured.

## Architecture

This is a tool to download labeled image datasets from the Hikvision AI Open Platform and convert them to COCO format. Python 3.8+, Windows-first with tkinter GUI.

### Module map

| Module | Role |
|--------|------|
| `main.py` | CLI entry point, argparse dispatcher (GUI / auto / manual / export-coco) |
| `core/__init__.py` | Package exports: `COCOConverter`, `ConversionResult` only |
| `core/auth.py` | `AuthManager` — holds token+cookies, builds API headers |
| `core/api_client.py` | `HikvisionAPIClient` — all REST calls to `https://ai.hikvision.com/api/saas/ai-training/algorithms`, paginated image listing, batch annotation fetching |
| `core/downloader.py` | `DatasetDownloader` — three-phase download (list→batch annotate→concurrent download), AES-ECB image decryption |
| `core/converter.py` | `COCOConverter` — two-pass: collects unique `label_name`s as categories, then builds COCO `images[]`/`annotations[]` |
| `browser/bb_browser_bridge.py` | `BBBrowserBridge` — CDP over `http://localhost:9222`, extracts `dataset_id`/`version_id` from page URL, two-tier cookie acquisition |
| `gui/main_window.py` | `tkinter` GUI with background threading for download/export, `config.json` persistence |
| `utils/` | Empty directory (reserved for future use) |
| `decrypt_existing_images.py` | Standalone script — bulk AES-ECB decrypt encrypted images in place using keys from annotation JSONs |
| `fetch_annotations_backup.py` | Legacy script — manual API exploration with hardcoded credentials |

### Cookie acquisition flow

The `BBBrowserBridge.get_cookies()` uses a two-tier strategy:
1. **Primary**: `browser_cookie3.chrome()` reads cookies from local Chrome cookie DB (no WebSocket needed, works with Chrome 111+).
2. **Fallback**: WebSocket CDP `Network.getCookies` (requires `bb-browser` or `--remote-allow-origins`).

### Download pipeline (three-phase)

`DatasetDownloader.download_all()` runs in three sequential phases:

1. **List images**: `HikvisionAPIClient.get_all_images(is_tagged=1)` paginates through the API 100 items per page, yielding `ImageFile` objects (only tagged/labeled images).
2. **Batch fetch annotations**: Accumulates all `file_id`s, then calls `get_annotations(batch_ids)` in batches of 50 (API limit) via POST `/datasets/files/targets/query`.
3. **Concurrent download**: `ThreadPoolExecutor` with `max_workers=5` downloads images in parallel via `requests.get`, then writes per-image annotation JSONs synchronously. Skips if both image and annotation already exist (idempotent).

### Image download

- Images are downloaded via raw `requests.get` (not aiohttp — the README is outdated on this point).
- If the `ImageFile` has a `key` field, the response body is AES-ECB decrypted (`pycryptodome`) after Base64 decoding.
- Files < 100 bytes after download are treated as corrupt and deleted.
- `_sanitize_filename()` strips Windows-invalid characters (`<>:"/\|?*`).

### COCO conversion

`COCOConverter.convert()` expects a directory with `images/` and `annotations/` subdirs (the download output structure). It:
1. Scans all `.json` files in `annotations/`.
2. Pass 1: collects unique `label_name` values, assigns `category_id` by first-seen order. **category_id starts at 1** (`len(category_map) + 1`), but `category_map.get(label, 0)` defaults to 0 for unmapped labels — unannotated labels silently get an invalid id.
3. Pass 2: matches each annotation JSON to its image file by stem, builds COCO `images[]` and `annotations[]` with `segmentation` polygons derived from bbox. Supports bbox in dict (`xmin/ymin/xmax/ymax`, also `x1/y1/x2/y2`, `left/top/right/bottom`) or list (`[xmin, ymin, w, h]`) format. Falls back to PIL for image dimensions if annotation JSON has missing w/h.
4. Copies images into `COCO/images/`, writes `COCO/annotations/instances.json`.
5. Skips annotations with empty labels or invalid bboxes (w <= 0 or h <= 0).

### Output structure

```
dataset_{id}_{timestamp}/
├── images/          # Original images
├── annotations/     # Per-image Hikvision JSONs
└── COCO/            # Generated by --export-coco
    ├── images/      # Image copies
    └── annotations/instances.json
```

### config.json

Persists `token`, `dataset_id`, `version_id` from the last successful connection. Loaded automatically at GUI startup to restore session. Listed in `.gitignore` (contains secrets).

## Notes

### Unused dependencies

`requirements.txt` lists `aiohttp`, `aiofiles`, and `PyYAML`, but none are imported anywhere in the project. The actual runtime dependencies are: `requests` (HTTP), `websocket-client` (CDP), `browser-cookie3` (cookie DB reader), `pycryptodome` (AES decryption).

### Testing

No formal test framework or `tests/` directory. Each core module has an inline `if __name__ == "__main__"` block with a manual smoke-test function that uses live credentials:

| File | Function |
|------|----------|
| `core/auth.py` | `test_auth()` |
| `core/api_client.py` | `test_api_client()` |
| `core/downloader.py` | `test_downloader()` |
| `browser/bb_browser_bridge.py` | `test_bridge()` |

These are not automated — run each file directly to smoke-test with a real Hikvision connection.
